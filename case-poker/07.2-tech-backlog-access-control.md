## Architecture Decisions (ADRs)

| # | Decision | Rationale | CTO sign-off |
|---|---|---|---|
| ADR-001 | Participant state machine added to existing participant record — not a separate service | Session-scoped. A new service adds operational overhead at current scale. | ✓ |
| ADR-002 | Anonymous aliases assigned server-side by join order (`Participant A`, `B`, etc.) — deterministic per session | Prevents alias collision within a session. Real name never included in participant-targeted payloads. | ✓ |
| ADR-003 | OIDC `groups` claim requested in existing OAuth2 middleware for Azure AD tenants — validated at room join, not at login | Avoids a parallel auth system. Just-in-time resolution per room. Non-Azure tenants unaffected. | ✓ |
| ADR-004 | LGPD routing implemented as a per-tenant write flag (`lgpd_resident: true`) — participant data for flagged tenants writes to `sa-east-1` RDS endpoint | Option C from CTO assessment. Additive config flag, zero impact to existing tenants. | ✓ |
| ADR-005 | Invite tokens: `crypto.randomBytes(16)` (128-bit), expire on first use or session end | Cryptographically secure, brute-force resistant. Validated server-side at join. | ✓ |
| ADR-006 | Removed participant WebSocket connection terminated server-side immediately — `participant_removed` close reason sent before disconnect | Prevents continued event reception after removal. Client-side enforcement alone is insufficient. | ✓ |

## Definition of Done

A story is done when **all** of the following are true:

- [ ] All acceptance criteria defined in the Product Backlog story are met and verified by QA
- [ ] Server-side enforcement confirmed for all security-sensitive behavior — no client trust
- [ ] Unit tests cover the happy path and all story edge cases
- [ ] Code reviewed and approved by Tech Lead
- [ ] All ADRs reviewed and signed off by CTO
- [ ] No regressions in existing session flow — QA regression suite passes
- [ ] For ST-009: integration test passed with Construtora Ágil test tenant before story is closed
- [ ] For ST-010: data residency validated in staging — no LGPD tenant data in `us-east-1`
- [ ] Tech Lead closes the story in the backlog

## Rollout Strategy

1. **Feature flag** `feature_access_control` — disabled by default for all tenants on deploy
2. **Infrastructure gate** — `sa-east-1` RDS endpoint provisioned and confirmed by CTO before flag is enabled for any LGPD tenant
3. **Azure AD gate** — OIDC integration enabled only for tenants with confirmed Azure AD configuration; Construtora Ágil must complete Azure portal registration first
4. **Internal validation** — flag enabled on internal test account; Tech Lead + QA run all 3 access modes, anonymous mode, removal, and role assignment scenarios
5. **Construtora Ágil staging** — flag enabled on client staging tenant; client IT confirms LGPD data residency and Azure AD role mapping
6. **Client LGPD sign-off** — Construtora Ágil IT (Fernanda Ramos) provides written confirmation that data residency requirement is met before production release
7. **General availability** — flag enabled for all tenants; LGPD routing active only for `lgpd_resident: true` tenants
8. **Rollback** — disable `feature_access_control` flag; all schema changes are additive — no rollback migration required; LGPD routing disabled by toggling tenant flag
