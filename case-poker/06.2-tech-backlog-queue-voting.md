## Decisões de Arquitetura (ADRs)

| # | Decisão | Justificativa |
|---|---|---|
| ADR-001 | Estado da fila armazenado como array JSON ordenado no registro de sessão, não em tabela separada | Escopo de sessão, limitado (máx. 100 itens). Evita overhead de join. Migração mais simples — campo aditivo no registro existente. |
| ADR-002 | Valores de votos excluídos dos payloads de broadcast WebSocket até o evento `votes_revealed` — aplicado server-side | Confiança no cliente é bypassável via DevTools. O servidor é o único ponto de aplicação seguro. Payload envia apenas `{participant_id, status: "voted"}` até a revelação. |
| ADR-003 | `item_revealed`, `votes_revealed`, `item_skipped` adicionados ao event bus existente — sem novo broker | Eventos de baixa frequência, com escopo de sessão. Infraestrutura existente absorve sem preocupação de escalabilidade. |
| ADR-004 | Reconexão do facilitador: snapshot do estado da sessão na desconexão, restaurado dentro do período de graça de 5 min | Previne perda de estado sem distributed lock. Após 5 min, a sessão encerra limpa com notificação aos participantes. |

## Definição de Pronto (DoD)

Uma história está pronta quando **todos** os itens abaixo são verdadeiros:

- [ ] Todos os critérios de aceite definidos na história do Product Backlog foram atendidos e verificados pelo QA
- [ ] Ocultação de votos confirmada server-side — nenhum valor de voto presente nos payloads WebSocket de participantes antes de `votes_revealed` (verificado via inspeção de rede)
- [ ] Testes unitários cobrem o happy path e todos os edge cases definidos na história
- [ ] Código revisado e aprovado pelo Tech Lead
- [ ] Sem regressões no fluxo de sessão existente — regression suite do QA aprovado
- [ ] Eventos de telemetria confirmados recebidos no pipeline de staging
- [ ] Tech Lead encerra a história no backlog

## Estratégia de Rollout

1. **Feature flag** `feature_queue_voting` — desabilitada por padrão para todos os tenants no deploy
2. **Validação interna** — flag habilitada na conta de teste interna; PM + CS executam simulação completa de cerimônia end-to-end
3. **Acesso antecipado Banco Meridional** — flag habilitada para o Banco Meridional 5 dias úteis antes do release geral; CS acompanha a primeira cerimônia ao vivo
4. **Disponibilidade geral** — flag habilitada para todos os tenants após Banco Meridional confirmar sem problemas
5. **Rollback** — desabilitar flag `feature_queue_voting`; mudanças de schema são aditivas e nullable — nenhuma migração de rollback necessária
