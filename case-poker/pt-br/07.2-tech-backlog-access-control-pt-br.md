## Decisões de Arquitetura (ADRs)

| # | Decisão | Justificativa | Sign-off do CTO |
|---|---|---|---|
| ADR-001 | Máquina de estado do participante adicionada ao registro de participante existente — não um serviço separado | Com escopo de sessão. Um novo serviço adiciona overhead operacional na escala atual. | ✓ |
| ADR-002 | Aliases anônimos atribuídos server-side por ordem de join (`Participante A`, `B`, etc.) — determinístico por sessão | Previne colisão de alias dentro de uma sessão. Nome real nunca incluído em payloads direcionados a participantes. | ✓ |
| ADR-003 | Claim OIDC `groups` solicitado no middleware OAuth2 existente para tenants Azure AD — validado no join da sala, não no login | Evita um sistema de auth paralelo. Resolução just-in-time por sala. Tenants não-Azure não são afetados. | ✓ |
| ADR-004 | Roteamento LGPD implementado como flag de escrita por tenant (`lgpd_resident: true`) — dados de participantes de tenants flagados são gravados no endpoint RDS `sa-east-1` | Opção C da avaliação do CTO. Flag de config aditiva, zero impacto em tenants existentes. | ✓ |
| ADR-005 | Tokens de convite: `crypto.randomBytes(16)` (128-bit), expiram no primeiro uso ou fim da sessão | Criptograficamente seguro, resistente a força bruta. Validado server-side no join. | ✓ |
| ADR-006 | Conexão WebSocket do participante removido encerrada server-side imediatamente — razão de fechamento `participant_removed` enviada antes da desconexão | Previne continuação de recebimento de eventos após remoção. Aplicação somente client-side é insuficiente. | ✓ |

## Definição de Pronto (DoD)

Uma história está pronta quando **todos** os itens abaixo são verdadeiros:

- [ ] Todos os critérios de aceite definidos na história do Product Backlog foram atendidos e verificados pelo QA
- [ ] Aplicação server-side confirmada para todos os comportamentos com sensibilidade de segurança — sem confiança no cliente
- [ ] Testes unitários cobrem o happy path e todos os edge cases da história
- [ ] Código revisado e aprovado pelo Tech Lead
- [ ] Todos os ADRs revisados e aprovados pelo CTO
- [ ] Sem regressões no fluxo de sessão existente — regression suite do QA aprovado
- [ ] Para ST-009: teste de integração aprovado com tenant de teste da Construtora Ágil antes de a história ser fechada
- [ ] Para ST-010: residência de dados validada em staging — nenhum dado de tenant LGPD em `us-east-1`
- [ ] Tech Lead encerra a história no backlog

## Estratégia de Rollout

1. **Feature flag** `feature_access_control` — desabilitada por padrão para todos os tenants no deploy
2. **Gate de infraestrutura** — endpoint RDS `sa-east-1` provisionado e confirmado pelo CTO antes da flag ser habilitada para qualquer tenant LGPD
3. **Gate Azure AD** — integração OIDC habilitada apenas para tenants com configuração Azure AD confirmada; Construtora Ágil deve completar o registro no portal Azure primeiro
4. **Validação interna** — flag habilitada na conta de teste interna; Tech Lead + QA executam todos os 3 modos de acesso, modo anônimo, remoção e cenários de atribuição de papel
5. **Staging Construtora Ágil** — flag habilitada no tenant de staging do cliente; TI do cliente confirma residência de dados LGPD e mapeamento de papel Azure AD
6. **Sign-off LGPD do cliente** — TI da Construtora Ágil (Fernanda Ramos) fornece confirmação por escrito de que o requisito de residência de dados está atendido antes do release em produção
7. **Disponibilidade geral** — flag habilitada para todos os tenants; roteamento LGPD ativo apenas para tenants `lgpd_resident: true`
8. **Rollback** — desabilitar flag `feature_access_control`; todas as mudanças de schema são aditivas — nenhuma migração de rollback necessária; roteamento LGPD desabilitado pelo toggle da flag do tenant
