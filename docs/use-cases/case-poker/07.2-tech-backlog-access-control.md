## Architectural Decisions (ADRs)

| # | Decision | Justification | CTO Sign-off |
|---|---|---|---|
| ADR-001 | Participant state machine added to the existing participant record — not a separate service | Session-scoped. A new service adds operational overhead at current scale. | ✓ |
| ADR-002 | Anonymous aliases assigned server-side by join order (`Participant A`, `B`, etc.) — deterministic per session | Prevents alias collision within a session. Real name never included in payloads directed at participants. | ✓ |
| ADR-003 | OIDC `groups` claim requested in the existing OAuth2 middleware for Azure AD tenants — validated at room join, not at login | Avoids a parallel auth system. Just-in-time resolution per room. Non-Azure tenants are not affected. | ✓ |
| ADR-004 | LGPD routing implemented as a per-tenant write flag (`lgpd_resident: true`) — participant data from flagged tenants is written to the `sa-east-1` RDS endpoint | Option C from the CTO's assessment. Additive config flag, zero impact on existing tenants. | ✓ |
| ADR-005 | Invite tokens: `crypto.randomBytes(16)` (128-bit), expire on first use or session end | Cryptographically secure, brute-force resistant. Validated server-side on join. | ✓ |
| ADR-006 | Removed participant's WebSocket connection closed server-side immediately — close reason `participant_removed` sent before disconnection | Prevents continued event receipt after removal. Client-only enforcement is insufficient. | ✓ |

## Definition of Done (DoD)

A story is done when **all** of the following are true:

- [ ] All acceptance criteria defined in the Product Backlog story have been met and verified by QA
- [ ] Server-side enforcement confirmed for all security-sensitive behaviors — no client trust
- [ ] Unit tests cover the happy path and all edge cases in the story
- [ ] Code reviewed and approved by the Tech Lead
- [ ] All ADRs reviewed and approved by the CTO
- [ ] No regressions in the existing session flow — QA regression suite passed
- [ ] For ST-009: integration test passed with Construtora Ágil test tenant before the story is closed
- [ ] For ST-010: data residency validated in staging — no LGPD tenant data in `us-east-1`
- [ ] Tech Lead closes the story in the backlog

## Rollout Strategy

1. **Feature flag** `feature_access_control` — disabled by default for all tenants on deploy
2. **Infrastructure gate** — `sa-east-1` RDS endpoint provisioned and confirmed by CTO before the flag is enabled for any LGPD tenant
3. **Azure AD gate** — OIDC integration enabled only for tenants with confirmed Azure AD configuration; Construtora Ágil must complete registration in the Azure portal first
4. **Internal validation** — flag enabled on internal test account; Tech Lead + QA run all 3 access modes, anonymous mode, removal, and role assignment scenarios
5. **Construtora Ágil staging** — flag enabled on the client staging tenant; client IT confirms LGPD data residency and Azure AD role mapping
6. **Client LGPD sign-off** — Construtora Ágil IT (Fernanda Ramos) provides written confirmation that the data residency requirement is met before production release
7. **General availability** — flag enabled for all tenants; LGPD routing active only for `lgpd_resident: true` tenants
8. **Rollback** — disable `feature_access_control` flag; all schema changes are additive — no rollback migration needed; LGPD routing disabled by tenant flag toggle
