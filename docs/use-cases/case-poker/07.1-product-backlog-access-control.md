## EP-001 — Access Modes

**Objective:** Give the room owner control over who can enter a session, replacing the current open-link model.

---

### ST-001 — Open mode (current behavior preserved)

**As** a room owner,
**I want** the current open-link behavior to remain the default,
**so that** existing rooms and users are not disrupted by this change.

**Acceptance Criteria:**
- [ ] Rooms without an explicit access mode configured behave exactly as they do today
- [ ] No existing user is required to take any action due to this release

**Edge Cases:**
- [ ] If an existing room is opened after this release without an access mode configured, it continues working without prompting the owner to configure access
- [ ] If a participant joins an open room while a migration is in progress, the join succeeds — the migration does not cause the join to fail

---

### ST-002 — Invite-only access

**As** a room owner,
**I want** to generate invite links so only people I have invited can enter,
**so that** I control exactly who participates in the session.

**Acceptance Criteria:**
- [ ] I can generate one or more invite links via the room settings panel
- [ ] Only someone with a valid invite link can enter the room
- [ ] Someone without an invite who tries to enter via the room URL sees: "This room requires an invite."
- [ ] I can revoke an invite link before it has been used
- [ ] Used invite links cannot be reused

**Edge Cases:**
- [ ] If someone uses a revoked invite link, they see: "This invite link is no longer valid."
- [ ] If someone tries to reuse an already-used link, they see: "This invite link has already been used."
- [ ] If I switch the room from Invite-only to Open after sending links, people with links can still enter (now via open mode) — existing tokens are invalidated
- [ ] If I generate 10+ invite links and the panel becomes long, it remains scrollable and usable — no UI overflow
- [ ] If the same person uses two different invite links in rapid succession (race condition), only one session is created — not two

---

### ST-003 — Mandatory approval access

**As** a room owner,
**I want** to approve or deny join requests in real time,
**so that** I decide who enters on a case-by-case basis without distributing invite links.

**Acceptance Criteria:**
- [ ] Anyone with the room URL can submit a join request
- [ ] I receive an immediate notification when someone requests to join: "User X is requesting entry"
- [ ] I can approve the request (they join immediately) or deny it (they see a denial message)
- [ ] If I do not respond within 5 minutes, the request expires and the person sees: "Request expired. Contact the room owner."
- [ ] Approved participants join with the default role (Voter) unless I assign otherwise

**Edge Cases:**
- [ ] If 5+ people request to join simultaneously, I see all pending requests in a queue — none are silently discarded
- [ ] If I am disconnected when a request arrives, it remains pending and I see it upon reconnecting (if within the expiration window)
- [ ] If I approve a request but the participant has already navigated away, they are marked as approved — if they return within the session, they join without re-requesting
- [ ] If the same person submits multiple requests in rapid succession (double-click), only one request is created
- [ ] If a denied participant immediately submits a new request, they can do so — the denial does not permanently block them from requesting again

## EP-002 — Anonymous Mode

**Objective:** Allow the room owner to hide participant identities from each other to support compliant ceremonies with mixed internal and external participants.

---

### ST-004 — Room owner activates anonymous mode

**As** a room owner,
**I want** to activate a mode where participants cannot see each other's real names,
**so that** I can run ceremonies with external contractors without exposing identities.

**Acceptance Criteria:**
- [ ] I can activate anonymous mode via the room settings panel before or during a session
- [ ] Once activated, anonymous mode cannot be turned off for the remainder of that session
- [ ] All participants are notified when anonymous mode is activated
- [ ] I always see real names regardless of whether anonymous mode is active

**Edge Cases:**
- [ ] If I accidentally click the anonymous mode toggle, a confirmation dialog prevents accidental activation: "Anonymous mode cannot be undone in this session. Activate?"
- [ ] If anonymous mode is activated while a vote is in progress, participants who already voted keep their vote — their display name changes to an alias immediately
- [ ] If a new participant joins after anonymous mode has been activated, they receive an alias immediately — they never see real names

---

### ST-005 — Participants see aliases in anonymous mode

**As** a participant in an anonymous session,
**I want** to see other participants as aliases (e.g., "Participant A", "Participant B") instead of names,
**so that** my identity and others' are not exposed during the ceremony.

**Acceptance Criteria:**
- [ ] Each participant in the room is represented by a consistent alias throughout the session
- [ ] The same person always appears with the same alias — it does not change during the session
- [ ] I never see another participant's real name in anonymous mode
- [ ] The facilitator can still see real names in their own view

**Edge Cases:**
- [ ] If a participant disconnects and reconnects, they receive the same alias they had before — not a new one
- [ ] If a participant is removed and a new participant joins, the new participant does not receive the removed participant's alias — aliases are not recycled within a session
- [ ] If all 26 single-letter aliases are exhausted (26+ participants), aliases continue with a defined extension pattern (e.g., "Participant AA") without breaking
- [ ] If a participant reloads the page, their alias is the same after reload — not reassigned

## EP-003 — Role Management

**Objective:** Allow the room owner to distinguish between participants who vote and those who observe, so that non-estimating participants do not influence results.

---

### ST-006 — Room owner assigns Voter or Observer role

**As** a room owner,
**I want** to assign each participant as a Voter or Observer,
**so that** product managers or executives can participate without voting.

**Acceptance Criteria:**
- [ ] I can assign or change the role of any participant before the session starts or between items
- [ ] A Voter I demote to Observer during an item has their current vote invalidated
- [ ] I cannot promote an Observer to Voter after voting has already started for the current item
- [ ] The participant is notified when their role changes

**Edge Cases:**
- [ ] If I demote a Voter to Observer and immediately promote them back (before any vote), their vote input is re-enabled cleanly
- [ ] If I demote a Voter who has already voted, a confirmation is shown: "This will invalidate their current vote. Continue?"
- [ ] If a participant's role changes while they are disconnected, they see the updated role upon reconnecting
- [ ] The room owner's role cannot be changed by anyone, including by themselves

---

### ST-007 — Observer experience

**As** an Observer,
**I want** to follow the session in real time without any voting controls,
**so that** I can stay informed without accidentally influencing estimates.

**Acceptance Criteria:**
- [ ] I can see all session content — current item, votes after reveal, participant list
- [ ] I have no voting interface — no way to submit a vote even if I try
- [ ] If I am demoted from Voter to Observer during a session, my voting controls disappear immediately
- [ ] The facilitator sees my role labeled as "Observer" in the participant list

**Edge Cases:**
- [ ] If I am demoted to Observer in the middle of a vote submission (race condition), the vote is not accepted — I see a message: "Your role changed. Your vote was not submitted."
- [ ] If I try to interact with voting controls via browser developer tools, the server silently rejects the submission and my Observer status remains unchanged
- [ ] If I am an Observer in anonymous mode, I still appear as an alias to other participants — not as "Observer [real name]"

## EP-004 — Participant Removal

**Objective:** Allow the room owner to remove a participant from an active session when needed.

---

### ST-008 — Room owner removes a participant

**As** a room owner,
**I want** to remove a participant from an active session at any time,
**so that** I can manage access and resolve situations where someone should not be in the room.

**Acceptance Criteria:**
- [ ] I can remove any participant at any time during an active session
- [ ] The removed participant immediately sees: "You have been removed from this session."
- [ ] Any vote the removed participant submitted for the current item is invalidated
- [ ] The removed participant cannot re-enter the same session after being removed

**Edge Cases:**
- [ ] I cannot remove myself (the room owner) — the remove action does not appear on my own participant entry
- [ ] If I remove a participant while votes are being revealed (race condition), the removal succeeds but their vote — if already included in the reveal payload — is still shown in the current reveal and invalidated from the next item onward
- [ ] If a removed participant tries to re-enter using an invite link generated before the removal, the link is invalid for this session — they see the removed state message
- [ ] If I accidentally remove a participant, there is no undo — they must request to join again (if the room is in approval mode) or receive a new invite link

## EP-005 — Azure AD Integration

**Objective:** Allow Construtora Ágil participants to join rooms with their roles automatically assigned by their Azure AD groups, without manual configuration by the room owner.

> External dependency: this epic requires Construtora Ágil's IT team to register the platform as an approved application in the Azure AD tenant and provide the tenant ID. Must be completed by 2026-04-14 for the delivery timeline to hold.

---

### ST-009 — Roles are automatically assigned by Azure AD group membership

**As** a Construtora Ágil employee or contractor,
**I want** my role on the platform to be set automatically when I join a room based on my Azure AD group,
**so that** the room owner does not need to manually configure who is a Voter and who is an Observer.

**Acceptance Criteria:**
- [ ] When I join a room as a member of the Azure AD group `pokerplan-voters`, I receive the Voter role automatically
- [ ] When I join as a member of `pokerplan-observers`, I receive the Observer role automatically
- [ ] If I do not belong to any group, I cannot enter and I see: "Your account is not authorized for this room."
- [ ] This behavior only applies to rooms of tenants with Azure AD configured — other tenants are not affected

**Edge Cases:**
- [ ] If I belong to both `pokerplan-voters` and `pokerplan-observers` simultaneously, the Voter role takes precedence — the conflict resolution rule is documented and consistent
- [ ] If the Azure AD `groups` claim is missing or empty in the token (e.g., misconfiguration on the client side), I am blocked with a clear message: "Could not verify your authorization. Contact your IT administrator."
- [ ] If the Azure AD service is temporarily unavailable when I try to join, I see: "Authorization service unavailable. Please try again in a moment." — the platform does not fail open and grant me access
- [ ] If my Azure AD group membership changes between sessions (e.g., moved from Voters to Observers by IT), my role reflects the updated group on my next join — not the previously cached role

## EP-006 — LGPD Data Residency

**Objective:** Ensure that Construtora Ágil participant identity data is stored in Brazil, meeting their legal data governance requirements under the LGPD.

> Infrastructure prerequisite: the database endpoint in the Brazil region (`sa-east-1`) must be provisioned by the CTO before this epic can be delivered. Target: 2026-05-05.

---

### ST-010 — Participant data from LGPD-flagged tenants is stored in Brazil

**As** Construtora Ágil,
**I need** all participant session data — names, emails, and session records — to be stored in Brazil,
**so that** the platform meets our data governance obligations under the LGPD.

**Acceptance Criteria:**
- [ ] Participant session data from our account is stored exclusively in the Brazil region (`sa-east-1`)
- [ ] No participant data from our account is stored in any other region
- [ ] This is confirmed in writing by the platform team before we go into production
- [ ] Other tenants' data storage is not affected by this change

**Edge Cases:**
- [ ] If the `sa-east-1` database endpoint is temporarily unavailable, the session join fails with a clear error — the platform does not fall back to `us-east-1` and silently store data outside Brazil
- [ ] If a participant's data is partially written before a connection failure during join, the partial record is rolled back — no incomplete records in any region
- [ ] If the tenant LGPD flag is incorrectly disabled after go-live (misconfiguration), a monitoring alert fires and data written to the wrong region is flagged for immediate review

## EP-007 — Room Owner Settings

**Objective:** Provide a single, coherent settings panel where the room owner can configure all access and anonymity options.

---

### ST-011 — Room owner manages all access settings from one panel

**As** a room owner,
**I want** a clear settings panel where I can configure how my room handles access and participant visibility,
**so that** I do not need to go to multiple places to set up a ceremony.

**Acceptance Criteria:**
- [ ] The settings panel is accessible from the room dashboard before the session starts and during an active session
- [ ] I can select the access mode (Open / Invite-only / Mandatory approval) from the panel
- [ ] When Invite-only is selected, I can generate and revoke invite links from the same panel
- [ ] I can activate anonymous mode from the panel, with a clear warning that it cannot be undone during the session
- [ ] All settings take immediate effect — no page reload required

**Edge Cases:**
- [ ] If I change the access mode while participants are actively joining, in-progress join attempts that started under the previous mode are resolved under that previous mode — not silently rejected
- [ ] If I change from Mandatory approval to Open while pending requests exist, all pending requests are automatically approved — not left indefinitely waiting
- [ ] If I change from Invite-only to Mandatory approval, previously generated invite links are invalidated — users trying to join with them are redirected to the approval request flow
- [ ] If the settings panel fails to load (network error), I see a clear error state — not a blank panel that silently ignores my changes
